📖 WebSocket Capacity Testing Guide

Quick Start

Standalone Mode (No Server):
Just open this file in a browser. Click test buttons to measure WebSocket provider capacity directly.

Dual Provider Mode (With Server):
Run your app server, then navigate to the dashboard for advanced metrics including load balancing and rotation.

Test Scenarios

New Tokens: Measures pump.fun discovery rate via subscribeNewToken
Trades (50/150): Stress tests trade monitoring with configurable token counts
Whales (20): Tests subscribeAccountTrade with fake whale wallets
Stress Test: Combines all three at max capacity

Key Metrics

Messages: Total count received from provider
Latency: Average message processing time (ms)
Throughput: Messages per second (msg/sec)
Overhead: Utilization % (red >80%, yellow >50%)

Expected Results

Healthy: Overhead <50%, latency <10ms, no errors
Capacity Found: Overhead hits 80%+, latency spikes >50ms
Practical Limits: Document max tokens/whales at <50% overhead

Architecture

Server Files:
• server/pumpfun-websocket-dual.ts - Dual-provider WebSocket manager
• server/websocket-capacity-tracker.ts - Overhead monitoring & load balancing

Test Files:
• websocket-capacity-test.html - This file (all-in-one)
• test-dual-websocket-capacity.html - Server-integrated dashboard

Rate Limits

PumpPortal: 15 connections max, 200 subs/sec, 5000 addresses/message
PumpDev: Same limits (independent connection)
Safe batching: 100 tokens every ~667ms (150 subs/sec)

Tips

  • Run tests for 5-10 minutes to see real throughput patterns
  • Monitor both providers simultaneously to compare performance
  • Note the overhead % where system becomes unstable
  • Use results to configure top-tier signal limits
  • Open DevTools (F12) to inspect actual WebSocket frames
